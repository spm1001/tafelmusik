The clearest articulation of why comments matter came from trying to use /close: Sameer wanted to react to specific parts of Claude's Orient reflection but had to paste-and-interleave or use ambiguous number references. The friction is the feature request. Comments aren't about code review or editorial workflow — they're about enabling inline reactions to specific text, from either party, without breaking the flow of the document. This reframes tfm-melemu: the UI affordance (a side pane linked to text selections) matters more than the backend (StickyIndex + Y.Map already exist). The dependency chain is: clean up the editor (tfm-kosotu) → repurpose the split pane as a comments sidebar → wire to the Y.Map backend (tfm-melemu).
