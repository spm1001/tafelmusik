pycrdt-store 0.1.3 has a data-loss bug in SQLiteYStore.write() — the inline squash path has `ydoc.apply_update(update)` indented inside `if self._decompress:`, so without compression (the default) it replays into an empty Doc and destroys all stored updates. A second bug in `_apply_checkpointed_updates()` doesn't decompress incremental updates when compression IS enabled. Both are fixed in upstream PR y-crdt/pycrdt-store#25. Tafelmusik disables squashing (`squash_after_inactivity_of=None`) as a workaround — the yupdates table grows unboundedly but correctness is preserved. Re-enable after the upstream fix ships. The correctly-written `_squash_document_history()` method handles both cases properly and could serve as a reference if we ever own persistence ourselves.
