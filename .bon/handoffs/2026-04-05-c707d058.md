## For Claudes to come

When you intercept a library's internal message handling — wrapping `onmessage`, monkey-patching dispatchers, hooking into protocol layers — you're betting on implementation details, not contracts. The bet might be correct (ours was), but it carries a maintenance obligation: document the assumption explicitly, add a gotcha entry, and track the work that eliminates the dependency. Every interception should carry its own expiry date. The pattern that makes this sustainable: contain, document, track elimination. The pattern that doesn't: intercept, ship, forget.

Separately: when you replace targeted code (rebuild called from exactly two places) with "simpler" code (rebuild on every update), check whether you've understood why the original had that constraint. If you can't articulate the reason, you haven't understood it well enough to remove it. Simplification that loses intention is just entropy with better formatting.
