### Reflection
The bon's `--how` specified WS (0x01 JSON messages over existing WebSocket) but we used HTTP instead. The HTTP endpoints from tfm-jafebo already existed and gave request-response semantics for free. Building request-response over WS would have meant inventing a protocol (request IDs, response matching, timeouts) alongside endpoints that already worked. The divergence was the right call but wasn't recorded until the closing review — this is the design-to-bon fidelity gap appearing again.

The "what did we miss" review mid-session was valuable: caught the flush_doc asymmetry, the stale CLAUDE.md, and the misleading test docstrings before they shipped as traps for future Claudes. The triage table (reflection → consequence → remedy → timing) made the session-end actions concrete rather than hand-wavy.

## For Claudes to come

When migrating an intermediate layer (Y.Map → HTTP/SQLite), the code change is the easy part. The dangerous part is the documentation surface area. CLAUDE.md, understanding.md, bon briefs, test docstrings, tool descriptions, module docstrings, and skill instructions all encode assumptions about which system is active. Every surface a future Claude might read becomes a potential source of wrong assumptions. In this session, updating every documentation surface took longer than the code change itself. When planning a migration, budget explicitly for documentation sweep — it's not cleanup after the migration, it's part of the migration.

The triage table pattern (reflection → consequence → remedy → timing) is the right way to bridge /close reflections to actions. Free-form "here are my thoughts, separately here are my actions" loses the causal link. Each observation implies an equal and opposite action — name it inline, then sort by timing.
