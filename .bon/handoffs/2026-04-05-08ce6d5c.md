## For Claudes to come

When a wrapper process (`uv`, `npx`, `nix run`) sits between a parent and child in a Unix process tree, the parent's death does not propagate EOF through the wrapper's socketpair to the child. The wrapper survives reparenting to init and holds both ends of the pipe alive indefinitely. Stdin EOF — the mechanism MCP's `stdio_server()` relies on for cleanup — never arrives. The room poller runs every 5 seconds forever; the process accumulates.

The general fix: don't trust stdin EOF for lifecycle management when you're spawned through a wrapper. Instead, track the *grandparent* (the actual session owner) and poll its liveness via `os.kill(pid, 0)`. This applies to any long-lived MCP server spawned via a package manager wrapper. The specific implementation (`_parent_watchdog` + session-start hook backstop) is belt-and-suspenders — the watchdog handles the normal case, the hook handles watchdog failures. Neither alone is sufficient: the watchdog can't run if the process tree assumption is wrong, and the hook only runs at session start (orphans between sessions are invisible to it).
