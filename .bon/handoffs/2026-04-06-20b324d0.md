## For Claudes to come

When building on CC's filesystem artefacts (session files, PID files, config), treat them as eventually-consistent external systems, not reliable local state. CC rewrites `~/.claude/sessions/{PID}.json` during startup — a temporary session ID appears first, then the final (resumed) ID overwrites it within 1-2 seconds. The MCP handshake (`InitializedNotification`) completes *before* CC finishes its own initialization. Any code that reads the session file at MCP startup will see the wrong value.

The general pattern: poll for stability rather than reading once. Read, wait, read again, use the value only when it stops changing. And always design fallback paths — if the artefact doesn't match expectations, degrade gracefully (room fallback in our case) rather than failing hard. The cost of a fallback path is small; the cost of a hard dependency on CC's internal lifecycle is unbounded.

This applies beyond session files. CC's process tree (`claude → uv → python`) is an implementation detail that could change. The hook system, the session directory structure, the channel notification Zod schema — all are undocumented interfaces where CC is the authority and we are downstream consumers. Belt-and-suspenders (watchdog + hook backstop, session poll + room fallback) is the right posture for anything that depends on CC internals.
