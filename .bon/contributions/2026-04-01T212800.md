Session 2026-04-01 evening: architectural rethink conversation.

Sameer and Claude stepped back from individual bon items to question whether the overall architecture is right. The feeling: "welded several bits of car together" — each piece has a reason but the whole is brittle. y-websocket monkey-patched alongside custom heartbeats, pycrdt-store with squashing disabled, signal files as notification workaround, drift tracking for a problem that shouldn't exist.

Key conclusions from the conversation:

1. **Yjs/CRDT is likely overkill.** Solves N-party offline-first peer-to-peer merge; we have 2 parties, 1 server, always online. CM6's native `@codemirror/collab` (OT with central authority) or even a file-on-disk model could be simpler. BUT co-editing IS needed — Claude's batch edits are fast and Sameer can't see when Claude is working (out of TUI in the browser), so simultaneous editing of different sections will happen.

2. **Comments should be separate entities, not embedded in the CRDT.** Inspired by W3C Web Annotation Model. Comments are standalone resources in SQLite, referencing the document via TextQuoteSelector (quoted text + 30 chars prefix/suffix context). This solves: flush_doc wiping comments, comments not surviving server restarts, comments not being queryable across documents. Re-anchoring uses text search with fuzzy fallback, same pattern as Hypothesis at scale.

3. **Comments aren't just annotations — they're action-carrying messages.** Instructions ("read this doc"), requests ("search for where we discussed this"), subjective reactions ("this feels negative"), AI-expressed doubt ("not confident about this"). The W3C `motivation` field maps to this: commenting, questioning, assessing, linking, suggesting. Comments are a richer interaction channel than linear chat because they have spatial context, semantic type, threading, and persistence.

4. **The real north star is spatially-anchored semi-synchronous conversation.** The pain point: during /close, Claude produces long output, Sameer must compose a single linear response addressing scattered points. If Sameer could react inline to specific parts while Claude continues (or responds to earlier reactions), the conversation fans out spatially and converges as threads resolve. This is closer to PR review than document annotation.

5. **Architecture should be two independent layers.** Document layer (text sync — OT or CRDT, TBD) and comment layer (SQLite + HTTP API + WebSocket broadcast). Neither owns the other. Document doesn't know about comments. Comments reference documents but don't depend on document data structure.

6. **CodeMirror 6 is the right editor.** Survey of alternatives (Monaco, ProseMirror, Tiptap, Lexical, Milkdown, block editors) confirmed CM6 is best fit for markdown source editing. The question is what sits beneath it.

7. **Deep Research brief composed** for claude.ai to research: CM6 collab real-world implementations, annotation systems for mutable documents, human+AI collaborative editing prior art, comments as conversation protocol.

Anchoring fragility on early drafts: Sameer raised this concern. Resolution: for the primary use case (reacting to stable output), anchoring is fine. For volatile drafts, orphaned comments degrade gracefully — the quote carries context even when the anchor drifts. Quote-based anchoring is "good enough" because the quote itself IS the context.
