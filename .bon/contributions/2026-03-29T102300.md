The "same asyncio Task" constraint for aconnect_ws is about cancel scope lifecycle (entering/exiting the WebSocket context manager), not about I/O operations. anyio task group subtasks are separate asyncio Tasks (verified: anyio's _spawn calls create_task), but they can safely send/receive on a WebSocket opened by the parent task because send_bytes/receive_bytes don't enter or exit cancel scopes — they just write to the stream. The original cross-task bug was about Provider.start() creating its own cancel scope tree in a different task from aconnect_ws, not about individual I/O operations. Structured concurrency guarantees subtasks are cancelled before the parent's scope exits, so the cleanup order is well-defined. This distinction matters for anyone extending the sync protocol with additional concurrent tasks.
