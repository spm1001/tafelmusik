Push notification into Claude's model context from external events has exactly two working paths, both discovered empirically on 2026-04-01: (1) background Bash commands that exit produce `<task-notification>` tags injected into context — the "mousetrap" pattern; (2) `asyncRewake: true` hooks that exit with code 2 call `enqueuePendingNotification` which injects via `wrapInSystemReminder`. Channel notifications (`notifications/claude/channel`) are architecturally correct but delivery-broken in CC as of v2.1.89. FileChanged hooks are a dead end — `systemMessage` goes to a transient UI notification widget (`addNotification` with 5s timeout), never model context. The CC source at `~/Repos/claude-code/src/` reveals the full hook architecture including swarm infrastructure (teammate mailbox, SendMessage routing, coordinator mode) that mirrors aboyeur's design but is not yet publicly accessible.

The y-websocket library has a hardcoded 30-second `messageReconnectTimeout` that kills WebSocket connections during quiet editing. The root cause of the "connection blip during comment entry" bug was not network instability but this silence timer — our ASGI server doesn't echo awareness messages, so during periods with no CRDT edits the browser sees 30s of silence and disconnects. The fix (resetting `wsLastMessageReceived` every 10s) is a monkey-patch; the proper fix is replacing y-websocket with standalone sync code, mirroring what we already did on the Python side.
