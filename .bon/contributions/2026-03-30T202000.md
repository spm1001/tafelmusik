The CRDT duplication bug exposed a fundamental tension in the file-backed room design: every hydration from file creates NEW CRDT operations (new client ID, new clocks), so any peer holding old operations will merge both sets — duplicating content. The fix for idle duplication is room retention (keep file-backed rooms in memory so reconnection syncs the same ops), but this doesn't help across server restarts where the room is necessarily fresh. The root cause is that Yjs CRDTs merge by operation identity, not content identity — two inserts of "hello" from different client IDs produce "hellohello", not "hello". This means file hydration should be treated as a one-time seed, not a repeatable operation. SQLite persistence of CRDT state is the mechanism that preserves operation identity across room lifecycles within a single server process; file hydration is the escape hatch when no CRDT state exists yet.
